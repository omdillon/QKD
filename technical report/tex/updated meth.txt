\section{Simulation Architecture}
\label{sec:simarch}

\subsection{Software Development Methodology}
\label{subsec:dev}

The simulation was developed using an iterative and incremental software engineering methodology in Python, utilising the Qiskit \gls{sdk}, Matplotlib and NumPy libraries, and MATLAB to perform 3D surface plotting. This approach was chosen to mitigate development risk and ensure the mathematical validity of each \gls{qkd} protocol before introducing broader system features. By adopting this structured lifecycle, the logic errors inherent in translating quantum mechanical models into software could be isolated and resolved early in the development process, ensuring reliable system execution. Git version control, hosted via GitHub, was employed to track changes and provide rollback capabilities during feature additions.

Development started with the implementation and testing of the baseline \gls{bb84} protocol. Once this core logic was validated against theoretical expectations, the methodology was repeated to systematically integrate the \gls{b92} and \gls{e91} protocols. Following the successful verification of all three foundational protocols, the system was incrementally expanded through the addition of channel noise modelling for all three protocols and eavesdropper threat modelling specifically for \gls{bb84} and B92. This staged integration ensured that subsequent layers of complexity did not obscure underlying logical faults. 

\subsection{Modular Package Design}
\label{subsec:moddesign}

\begin{figure}
    \centering
    \includegraphics[width=0.7\linewidth]{images.methodology/system overview.png}
    \caption{Modularised System Architecture.}
    \label{fig:systemfiles}
\end{figure}

Implemented as a modular Python package (\texttt{QKDSimulator}) structured across thirteen source files, the system is functionally segmented, as depicted in \Cref{fig:systemfiles}. This modular architecture was a deliberate design decision driven not only by good programming practice but also by the need to enable both the independent operation of protocols and reproducibility in the configuration of experiments, as simulation parameters are decoupled from execution code.  

Following the initial validation of \gls{bb84}, the architecture was refined to employ an \gls{oop} paradigm, organised around a parent class, \texttt{QKDProtocol}, which defines the common interface shared by all three protocols. Each protocol child class (\gls{bb84}, \gls{b92}, \gls{e91}) overrides the parent class with independent operation logic while inheriting the common post-processing functions. To store the data of a protocol run, such as the sifted keys and \gls{qber}, a dataclass (\texttt{QKDResult}) encapsulates the outputs. This design ensures that the protocol data can be processed interchangeably by the benchmark module, ensuring uniform processing and reducing code duplication. 

\subsection{Experiment Configuration System}
\label{subsec:config}

Initially, simulation parameters were hardcoded to accelerate the implementation of the core protocol logic. However, as the project progressed to executing large-scale batch experiments with varying configurations, this approach became unscalable. To resolve this challenge, a dedicated configuration system was developed to load simulation parameters into the system, eliminating hardcoding. 

The initial iteration relied on parsing \texttt{.txt} files via basic string manipulation. As the volume of simulation parameters increased, this method proved illegible and error-prone, prompting a transition to the \texttt{.YAML} format. This shift enabled each parameter to be explicitly annotated with inline documentation.

During runtime, the selected configuration file is parsed into a \texttt{SimConfig} dataclass located in \texttt{config.py}, which validates the requested execution mode against the four supported modes: \texttt{single} (a single protocol test trial with fixed parameters), \texttt{sweep} (a parametric evaluation of channel Noise Strength), \texttt{eve\_sweep} (a parametric evaluation of eavesdropper interception rates), and \texttt{protocol\_comparison} (execution of multiple protocols under identical noise conditions). An example configuration snippet is provided in \cref{lst:yaml-config}.

\begin{lstlisting}[language=Python, caption={Example YAML Configuration for a \gls{bb84} Noise Sweep.}, label={lst:yaml-config}]
mode: sweep
protocol: bb84
n_\glspl{qubit}: 1000
n_trials: 20
noise_type: depolarizing
noise_min: 0.0
noise_max: 0.30
noise_steps: 30
output_dir: ./results/bb84_depolarizing
save_plots: true
\end{lstlisting}

\subsection{System Information Flow}
\label{subsec:infoflow}


\begin{figure}
    \centering
    \includegraphics[width=0.7\linewidth]{images.methodology/information flow.png}
    \caption{System Information Flow}
    \label{fig:pipeline}
\end{figure}

Building upon the configuration system, the execution of a simulation follows a deterministic, sequential information flow, as illustrated in \cref{fig:pipeline}. To maintain the logical separation of the system, the architecture is partitioned into four phases: Preparation, Measurement, Post-processing, and Plotting. These phases are also reflected in the modularisation of the Python package, simplifying the iterative development of the system.

During initial development, the computational overhead of simulating individual quantum circuits across large-scale sweeps ($>200\ runs$) became apparent through extensive runtimes. To optimise performance, the execution logic evolved to use batch processing. As depicted in the Measurement phase, instead of executing the \texttt{AerSimulator} iteratively for each \gls{qubit}, the system aggregates the prepared Qiskit circuits and dispatches them as a single batch job, mitigating the bottleneck and enabling the simulations to scale across parameter variations. The simple improvement is depicted below in \cref{lst:individual} and \cref{lst:batch}.

\begin{lstlisting}[language=Python, caption={Individual Circuit Processing.}, label={lst:individual}]
def _bob_measure(self, circuits: List[QuantumCircuit]):
        """Bob picks random bases and measures each \gls{qubit} sequentially."""
        self._bob_bases = np.random.randint(0, 2, size=self.n_\glspl{qubit})
        self._bob_results = np.zeros(self.n_\glspl{qubit}, dtype=int)
        
        for i, qc in enumerate(circuits):
            if self._bob_bases[i] == 1:
                qc.h(0)
            qc.measure(0, 0)
            
            job = self.backend.run([qc], shots=1, memory=True)            
            result = job.result()
            
            self._bob_results[i] = int(result.get_memory(0)[0])
\end{lstlisting}

\begin{lstlisting}[language=Python, caption={Batch Circuit Processing.}, label={lst:batch}]
def _bob_measure(self, circuits: List[QuantumCircuit]):
        """Bob picks random bases and measures each \gls{qubit} as a batch."""
        self._bob_bases = np.random.randint(0, 2, size=self.n_\glspl{qubit})
        self._bob_results = np.zeros(self.n_\glspl{qubit}, dtype=int)

        for i, qc in enumerate(circuits):
            if self._bob_bases[i] == 1:
                qc.h(0)
            qc.measure(0, 0)

        job = self.backend.run(circuits, shots=1, memory=True)
        result = job.result()
        for i in range(len(circuits)):
            self._bob_results[i] = int(result.get_memory(i)[0])
\end{lstlisting}

Importantly, the information flow accommodates channel interference via either noise created in the quantum backend, or as an eavesdropper. If specified in the configuration file, an \texttt{EveInterceptor} object is initialised during the Preparation phase. Crucially, Eve's interception acts as a modular interrupt within the standard \gls{pandm} pathway, enabling the threat model to be integrated without alteration to Bob's fundamental measurement knowledge.

Following batch execution, the system transitions to the classical Post-processing phase. While the majority of this pipeline is uniform across all configurations, the flow incorporates protocol-specific branching to perform post-processing calculations. Once all trials and sweep points conclude, the data undergoes statistical reduction to compute means and standard deviations. The results are then packaged into the benchmarking dataclass, visualised using Matplotlib, and written to the output directory. However, in the simulation of experiment 5 \& 6, which simulate the multivariable dynamics of \gls{bb84}, and \gls{b92}, the simulation data is exported via the\texttt{SurfaceBenchMarkData} object, which collects trial data during runtime, finally producing a \texttt{.csv} file for export.

Having established the overarching execution flow, the following sections detail the mathematical and operational implementation of the three \gls{qkd} protocols executed within this framework.

\subsection{BB84}
\label{subsec:bb84}

Serving as the foundational \gls{qkd} protocol, \gls{bb84} employs four encoding states through two mutually unbiased conjugate bases (rectilinear \& diagonal) to distribute cryptographic key material across a unidirectional quantum channel (\gls{qchan}) \cite{BB84}. Alice randomly selects both a bit value and a basis for each \gls{qubit}, producing one of the following states:

%
\begin{equation}
  \ket{0},\quad \ket{1},\quad
  \ket{+} = \frac{\ket{0}+\ket{1}}{\sqrt{2}},\quad
  \ket{-} = \frac{\ket{0}-\ket{1}}{\sqrt{2}}.
  \label{eq:bb84-states}
\end{equation}
%

 These states are transmitted to Bob over the \gls{qchan}, at which point Bob independently and randomly chooses a measurement basis for each incoming \gls{qubit}. As the bases are mutually unbiased, a measurement in the wrong basis yields a uniformly random outcome, providing no information about Alice's bit value \cite{QKDbible}. \Cref{fig:BB84operation} demonstrates an abstracted model of this process.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\linewidth]{images.methodology/bb84processTNR.png}
    \caption{BB84 Protocol Operation Model.}
    \label{fig:BB84operation}
\end{figure}

After transmission is complete, Alice and Bob switch to a pre-authenticated bidirectional classical channel (\gls{cc}) to perform key \gls{sifting}. Bob announces only his choice of basis for each \gls{qubit}, and Alice confirms which choices matched her preparation basis. All rounds in which the bases disagree are discarded. In an ideal, noiseless channel, the remaining \textit{sifted key} is perfectly correlated between Alice and Bob, and roughly $\sim50\%$ of the transmitted \glspl{qubit} survive the \gls{sifting} step.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\linewidth]{images.methodology/bb84QC.png}
    \caption{Qiskit Circuit Model for BB84.}
    \label{fig:bb84QC}
\end{figure}

To simulate transmission, Alice uses a NumPy pseudorandom number generator to select a bit ($a_i$) and basis ($\hat{a}_i$) for each round. As illustrated in the quantum circuit model \Cref{fig:bb84QC}, state encoding in Qiskit is achieved conditionally: an X gate flips the initial $\ket{0}$ state in the computational-basis to $\ket{1}$ if $a_i = 1$, followed by a Hadamard gate to rotate into the diagonal ($X$) basis if required. Crucially, an Identity gate (\texttt{qc.id(0)}) is appended immediately after preparation. This acts as the \gls{qchan}, providing a point to inject noise and eavesdrop without altering the logical quantum state.

For reception, Bob independently generates a measurement basis array ($\hat{b}_i$). If an $X$-basis measurement is chosen, a Hadamard gate is applied prior to the computational-basis measurement. Following circuit execution, basis arrays are compared, and the retained sifted key ($\sim50\%$ ideal conditions) is passed onto the post-processing pipeline, concluding the \gls{sifting} phase.

\subsection{B92}
\label{subsec:b92}


% theory
Bennett proposed a simplified protocol in 1992, building upon the critical insight that unconditional security simply requires that no measurement can perfectly distinguish between non-orthogonal quantum states \cite{ekertQI}. \gls{b92} achieves this through the encoding scheme in \Cref{eq:b92-states}.

%
\begin{equation}
  \ket{\varphi} =
  \begin{cases}
    \ket{0}  & \text{if } a = 0, \\[4pt]
    \ket{+}  & \text{if } a = 1.
  \end{cases}
  \label{eq:b92-states}
\end{equation}
%

\begin{figure}[h]
    \centering
    \includegraphics[width=0.7\linewidth]{images.methodology/b92basis.png}
    \caption{B92 Preparation \& Measurement Basis Angles.}
    \label{fig:b92basis}
\end{figure}
As illustrated in \Cref{fig:b92basis}, the encoding states are separated by $45^\circ$, validating the non-orthogonality criteria.

% implementation
Implementing Alice's preparation begins with initialising an empty single-\gls{qubit} quantum circuit, where if $a_i = 0$, the \gls{qubit} remains in the default $\ket{0}$ state. If $a_i = 1$, a Hadamard gate is applied to produce the $\ket{+}$ state. Preparation is concluded with the Identity gate (\texttt{qc.id(0)}) to provide the QC.

% Bobs is less intuitive, Bob is looking for a null result to take the complement.
Bob’s unintuitive measurement logic, as detailed in the operational flowchart in \Cref{fig:b92flowchart}, is the most distinctive feature of B92. Bob generates a random basis bit, $b_i$. If $b_i = 0$, he measures in the $Z$-basis ($\ket{0}, \ket{1}$). Alternatively, if $b_i = 1$, he applies a Hadamard gate to measure in the $X$-basis ($\ket{+}, \ket{+}$). Hence, due to the non-orthogonality principle, a measurement outcome is only conclusive if it logically contradicts one of Alice's two possible preparations.

Specifically, measuring a $\ket{0}$ state in the $Z$-basis will always yield an outcome of "0", making the result ambiguous. However, measuring a $\ket{+}$ state in the $Z$-basis yields the outcome "1" 50\% of the time. Therefore, an outcome of "1" in the $Z$-basis logically rules out the possibility that Alice prepared $\ket{0}$; enabling Bob to infer that Alice sent $\ket{+}$ (bit 1). Conversely, an outcome of "1" in the $X$-basis rules out the possibility that Alice prepared $\ket{+}$; Bob infers Alice sent $\ket{0}$ (bit 0).

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\linewidth]{images.methodology/b92 flowchart.png}
    \caption{B92 Algorithm Flowchart.}
    \label{fig:b92flowchart}
\end{figure}

% \gls{sifting}
Following circuit execution, the classical \gls{sifting} phase begins. Unlike \gls{bb84}, Alice and Bob do not exchange basis information. Instead, Bob publicly announces only the indices where his measurement outcome was a conclusive event ("1"). All inconclusive rounds ("0") are discarded as errors. For the subset of conclusive rounds, Bob’s final sifted key bit is calculated to be the logical complement of his measurement basis, $1 - b_i$.

Because \gls{b92} relies on only two states and discards ambiguous outcomes, its \gls{sifting} efficiency is fundamentally lower than that of \gls{bb84}, with a theoretical \gls{sifting} efficiency of $\sim25\%$ under ideal conditions, demonstrating the complexity-efficiency trade-off of the protocol. In practical deployment, this theoretically leaves the protocol more susceptible to channel noise, requiring longer transmission sequences to generate an equivalent key length to BB84.


\subsection{E91}
\label{subsec:e91}

Translating the Ekert protocol from abstract quantum mechanics to program logic proved to be challenging. \gls{e91} approaches \gls{qkd} from a fundamentally distinct approach in comparison to \gls{bb84} and \gls{b92}, exploiting quantum \gls{entanglement} to establish a shared secret key, with security guaranteed by violations of the Bell inequality rather than the quantum no-cloning theorem alone.

% theory
The \gls{e91} protocol is founded upon the generation of maximally \gls{entanglement} \gls{qubit} pairs in the singlet Bell state, $\ket{\Psi^-}$, defined as:

\begin{equation}
\ket{\Psi^-} = \frac{\ket{01} - \ket{10}}{\sqrt{2}}
\label{eq:e91_singlet}
\end{equation}

In this state, the measurement outcomes of Alice and Bob are perfectly anti-correlated, meaning if Alice measures $\ket{0}$, Bob necessarily measures $\ket{1}$, and vice versa, regardless of the measurement bases chosen, so long as the bases are aligned. This non-classical correlation cannot be reproduced by any local hidden-variable model, forming the security primitive of the protocol \cite{E91, Bell1964}.

The validation of security for this protocol is performed via the \gls{chsh} inequality \cite{CHSH1969}. The \gls{chsh} parameter $S$ is computed from correlation coefficients $E(a_i, b_j)$ measured across non-aligned angle pairs, defined as:

\begin{equation}
S = E(a_1, b_1) - E(a_1, b_3) + E(a_3, b_1) + E(a_3, b_3) \ ,
\label{eq:chsh_parameter}
\end{equation}

where $a_i$ and $b_j$ denote the measurement angles selected by Alice and Bob respectively, and $E(a_i, b_j)$ is the normalised correlation coefficient extracted from coincidence counts at angle pairing. Clauser \textit{et al.} went on to describe a value of $|S| = 2\sqrt{2}$ as the quantum mechanical maximum \cite{CHSH1969} of the parameter, with any significant decline towards $|S| \leq 2$ indicating the presence of an intercept threat.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\linewidth]{images.methodology/e91 XZ plane angles.png}
    \caption{E91 XZ-Plane Cross-section of Measurement Angles.}
    \label{fig:e91angles}
\end{figure}

As illustrated in \Cref{fig:e91angles}, the specific measurement angles employed by the simulator are chosen to satisfy the angular separations required by the \gls{chsh} test and to yield the two aligned pairs used for key generation.

\begin{figure}
    \centering
    \includegraphics[width=0.7\linewidth]{images.methodology/e91_circuit.png}
    \caption{E91 Quantum Circuit.}
    \label{fig:e91circuit}
\end{figure}

%\subsubsection*{State Preparation}
Due to the fact that the default two-\gls{qubit} Bell state generated by the standard Hadamard $\rightarrow$ CNOT sequence in Qiskit produces $\ket{\Phi^+} = \frac{1}{\sqrt{2}}(\ket{00} + \ket{11})$, not the required $\ket{\Psi^-}$ singlet, a practical challenge arises. To initialise the correct singlet state, the system logic appends an X gate followed by a Z gate to the first \gls{qubit} after the entangling operation, as described in \Cref{alg:e91_prep}, and illustrated in \Cref{fig:e91circuit}

\begin{algorithm}[H]
    \caption{E91 Singlet State Preparation.}\label{alg:e91_prep}
    \KwIn{Two-\gls{qubit} quantum circuit \texttt{qc}, \gls{qubit} indices \texttt{q[0]} (Alice), \texttt{q[1]} (Bob)}
    \KwOut{Circuit \texttt{qc} encoding $\ket{\Psi^-}$ across \texttt{q[0]} and \texttt{q[1]}}
    Apply Hadamard gate $H$ to \texttt{q[0]}\; \Comment{Initialise \texttt{q[0]} into an equal superposition}
    Apply CNOT gate with \texttt{q[0]} as control, \texttt{q[1]} as target\; \Comment{Entangle \texttt{q[0]} and \texttt{q[1]} to produce $\ket{\Phi^+}$}
    Apply X gate to \texttt{q[0]}\; \Comment{Flip computational basis of \texttt{q[0]} -- transforms $\ket{\Phi^+} \rightarrow \ket{\Psi^+}$}
    Apply Z gate to \texttt{q[0]}\; \Comment{Apply relative phase flip to \texttt{q[0]} -- transforms $\ket{\Psi^+} \rightarrow \ket{\Psi^-}$}
    \Return{\texttt{qc}}
\end{algorithm}

This four-step sequence is vital because the singlet $\ket{\Psi^-}$ possesses a relative phase of $-1$ between its $\ket{01}$ and $\ket{10}$ components. The Z gate on the first \gls{qubit} introduces this exact phase relationship, whilst the X gate performs the computational basis flip required to produce anti-correlated rather than correlated outcomes \cite{QiskitAer}. Selecting $\ket{\Psi^-}$ over $\ket{\Phi^+}$ is integral to the angular measurement decoding logic, as it ensures anti-correlation.

% measurement
Simulating measurement along an arbitrary angle $\theta$ in the XZ-plane, a rotation gate $R_y(-\theta)$ is applied to the relevant \gls{qubit} prior to $Z$-basis projection. This operation rotates the measurement basis so that the following computational-basis measurement is equivalent to a physical polarisation measurement at angle $\theta$ \cite{e91Qiskit}. \Cref{alg:e91_meas} presents this measurement procedure.

\begin{algorithm}[H]
    \caption{E91 Angular Measurement.}\label{alg:e91_meas}
    \KwIn{Circuit \texttt{qc} with $\ket{\Psi^-}$ prepared, angle sets $A_{\text{alice}} = \{0^\circ, 45^\circ, 90^\circ\}$, $A_{\text{bob}} = \{45^\circ, 90^\circ, 135^\circ\}$, shot count \texttt{n\_shots}}
    \KwOut{Measurement result list \texttt{outcomes[]}}
    \For{$i \leftarrow 1$ \KwTo \texttt{n\_shots}}{
        Sample angle $a$ randomly from $A_{\text{alice}}$\; \Comment{Alice randomly selects her measurement angle}
        Sample angle $b$ randomly from $A_{\text{bob}}$\; \Comment{Bob randomly selects his measurement angle}
        Apply $R_y(-a)$ to \texttt{q[0]}\; \Comment{Rotate Alice's state by $-a$ -- aligns measurement with chosen basis}
        Apply $R_y(-b)$ to \texttt{q[1]}\; \Comment{Rotate Bob's state by $-b$ -- aligns measurement with chosen basis}
        Measure \texttt{q[0]} $\rightarrow$ \texttt{c[0]}, \texttt{q[1]} $\rightarrow$ \texttt{c[1]}\; \Comment{Execute projective measurement in the $Z$-basis}
        Append $(a, b, \texttt{c[0]}, \texttt{c[1]})$ to \texttt{outcomes[]}\;
    }
    \Return{\texttt{outcomes[]}}
\end{algorithm}

With each shot executing independently against a freshly prepared singlet state, the statistical independence is preserved for \gls{chsh} correlation in post-processing.

% \gls{sifting}
Following the measurement phase, outcomes are categorised in line with the $3 \times 3$ \gls{sifting} Table, indexed by the nine possible Alice-Bob angle-pair combinations. Each combination is assigned to one of three categories, as shown in \Cref{tab:e91_sift}.

\begin{table}[h]
\centering
\caption{E91 \gls{sifting} Table. Key Pairs Contribute to the Raw Sifted Key; \gls{chsh} Pairs Estimate the Bell Parameter $S$; Discard Pairs are Rejected.}
\begin{tabular}{lccc}
\toprule
 & \textbf{Bob: $45^\circ$} & \textbf{Bob: $90^\circ$} & \textbf{Bob: $135^\circ$} \\
\midrule
\textbf{Alice: $0^\circ$} & \gls{chsh} & \gls{chsh} & Discard \\
\textbf{Alice: $45^\circ$} & Key & \gls{chsh} & \gls{chsh} \\
\textbf{Alice: $90^\circ$} & Discard & Key & Discard \\
\bottomrule
\end{tabular}
\label{tab:e91_sift}
\end{table}

Of the nine possible combinations, two pairs are designated for \textbf{Key Generation}, four pairs are designated for \textbf{CHSH estimation}, and three pairs are \textbf{discarded}. For the two pairs designated for key generation, Bob logically inverts his measurement outcomes ($1 - b_i$) to reconcile the anti-correlation of the singlet state and align the shared key. Since each of the nine combinations is equally probable, the theoretical \gls{sifting} efficiency is $2/9 \approx 22.2\%$, a trade-off inherent to the protocol's security verification overhead.

Initially implementing the \gls{sifting} Table as a series of conditional statements provided a necessary format to trace angle-pairing, despite the inefficient computational process. By adapting this process into a pre-compiled \gls{sifting} Table found in \Cref{tab:e91_sift}, the overhead was significantly reduced over large-scale simulation runs.

\section{QKD Network Topology}
\label{sec:networktopology}

Having described the protocol logic for \gls{bb84}, \gls{b92}, and \gls{e91}, it is important to distinguish the topologies through which they operate. Although each protocol ultimately enabled the secure distribution of cryptographic keys, the underlying architecture differs significantly between \gls{pandm} and \gls{eb} schemes, which in turn affects how the simulator must represent state distribution, channel noise, and eavesdropping. Crucially, across all network topologies, it is assumed that the classical channel is perfectly authenticated, and impervious to attack. While \gls{qkd} provides information-theoretic security for the quantum transmission, it fundamentally relies on this pre-existing classical authenticity to prevent attacks during post-processing.

\subsection{\gls{pandm} Topology}
\label{subsec:pmtopology}
In the \gls{pandm} topology, Alice acts as the sole quantum state source, encoding each \gls{qubit} locally before transmitting it to Bob over a unidirectional QC. Bob then performs a measurement on the received state, whilst the authenticated \gls{cc} is used only after transmission for post-processing stages. As depicted in \Cref{fig:pmtopology}, this architecture reflects the operational structure of both \gls{bb84} and \gls{b92}, in which a single logical quantum path connects preparation and detection.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\linewidth]{images.methodology/QuantumAbstractTNR.png}
    \caption{Abstract \gls{qkd} Network Topology of \gls{pandm} Protocols.}
    \label{fig:pmtopology}
\end{figure}

This topology directly corresponds to the simulation, since only one travelling \gls{qubit} must be modelled per round. Consequently, channel interference can be inserted at a single point between preparation and measurement, preserving a clear correlation between the abstract communication model and the implemented circuit flow.

\subsection{\gls{eb} Topology}
\label{subsec:ebtopology}
\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\linewidth]{images.methodology/EBtopology.png}
    \caption{Abstract \gls{qkd} Network Topology of \gls{eb} Protocols.}
    \label{fig:ebtopology}
\end{figure}

In the \gls{eb} topology, neither Alice nor Bob acts as the source of quantum information. Instead, an \gls{entanglement} photon source distributes one \gls{qubit} of an \gls{epr} pair to Alice and the other to Bob, after which both parties perform local measurements on their respective subsystems \cite{ekertQI}. As shown in \Cref{fig:ebtopology}, this replaces the unidirectional \gls{qchan} with a shared-resource model, in which security is inferred from measurement correlations rather than solely from the disturbance of transmitted states.

\section{Quantum Channel Interference}
\label{sec:channel_interference}

Ideal, noiseless conditions provide a useful baseline for protocol verification, but they do not reflect practical engineering environments. In any real-world \gls{qkd} deployment, systems face two distinct sources of quantum state disruption: environmental decoherence, naturally degrading quantum states during transmission, and malicious intervention, where an eavesdropper actively intercepts the channel. Accurately modelling these channel imperfections is therefore essential for a simulation platform that aims to evaluate protocol performance under realistic conditions.

\subsection{Depolarising Noise}
\label{subsec:noise}
% noise applie dto both arms of the e91 network, noise for alice and bob is the same
During physical transmission through optical fibre or free space, environmental factors cause \glspl{qubit} to lose their quantum coherence. A common abstract representation of this decoherence is the depolarising channel model. A depolarising channel replaces the original quantum state with a mixed state (white noise) with probability $p$, leaving the error-free state with probability $1-p$, providing a standard worst-case model of this isotropic noise \cite{QKDbible}. Mathematically, it maps an arbitrary single-\gls{qubit} state $\rho$ by applying a random Pauli error with total probability $p$:

\begin{equation}
\label{eq:depolarising}
\mathcal{E}_{dep}(\rho) = (1-p)\rho + \frac{p}{3}(X\rho X + Y\rho Y + Z\rho Z)
\end{equation}

Because the depolarising channel applies errors symmetrically across all axes of the Bloch sphere, it affects all encoding bases equally \cite{QKDbible}. Consequently, as the Noise Strength parameter $p$ increases, the probability of Bob measuring an erroneous outcome grows, directly increasing the Quantum Bit Error Rate (\gls{qber}).

To implement this without hardcoding errors into the protocol logic, the platform exploits Qiskit's \texttt{AerSimulator}. However, a challenge arises when applying noise to a circuit. By default, the noise models in Qiskit apply errors to specific gates across the entire circuit. If a depolarising error is assigned to a standard operation like the X or Hadamard gate, noise would corrupt the \gls{qubit} during Alice's preparation and Bob's measurement phases, effectively simulating an imperfect source/receiver, rather than a noisy \gls{qchan} (a particularly puzzling problem encountered in development).

Resolving this challenge required the implementation to leverage the Identity gate (\texttt{id}) to act as the QC. As described in \Cref{sec:protocol-implementation}, an \texttt{id} gate is added to the circuit before transmission. Binding the noise generation logic detailed in \Cref{alg:noise_model} to the \texttt{id} gate, a depolarising noise channel is achieved.

\begin{algorithm}[H]
    \caption{AerSimulator Backend Generation for Channel Noise.}\label{alg:noise_model}
    \KwIn{\texttt{noise\_type} (e.g. 'depolarizing'), \texttt{strength} $p$}
    \KwOut{Configured \texttt{AerSimulator} backend}
    \If{\texttt{noise\_type} $=$ 'none'}{
        \Return{default \texttt{AerSimulator()}} \Comment{Ideal channel -- no noise applied}
    }
    \texttt{noise\_model} $\leftarrow$ empty \texttt{NoiseModel()}\;
    \texttt{error} $\leftarrow$ generate \texttt{depolarizing\_error} with probability \texttt{strength}\; \Comment{Define depolarising error parameterized by $p$}
    Add \texttt{error} to \texttt{noise\_model}, targeting only \texttt{['id']} gates\; \Comment{Apply noise model exclusively to identity operations}
    \Return{\texttt{AerSimulator(noise\_model=noise\_model)}}
\end{algorithm}

By isolating the noise injection to the \texttt{id} gate, the simulator ensures preparation and measurement remain mathematically perfect, guaranteeing that any resultant \gls{qber} is a product of the simulated QC.

\subsection{Eavesdropping}
\label{subsec:eavesdropping}
While a noisy channel degrades quantum states passively, an eavesdropper represents an active adversarial threat. Due to the quantum no-cloning theorem \cite{WZ1982}, an eavesdropper cannot perfectly copy an unknown non-orthogonal quantum state. As such, the most fundamental attack against a \gls{qkd} system is the "intercept-resend" attack. In this threat model, Eve intercepts a \gls{qubit}, guesses a measurement basis, executes a measurement, and then attempts to conceal her presence by preparing and transmitting a new \gls{qubit} to Bob, encoded with her measurement result. Because Eve will inevitably guess the incorrect basis for a subset of the intercepted states, her presence introduces detectable errors that elevate the \gls{qber} beyond the noise level.
 
Within the simulator, this presence is encapsulated in the \texttt{EveInterceptor} class, with the intensity of attack governed by a configurable \texttt{interception\_rate} parameter ($0.0-1.0$), dictating the probability that a given state in the \gls{qubit} stream will be intercepted. \Cref{alg:eve_intercept} details the logical execution of the attack.

\begin{algorithm}[H]
    \caption{Intercept-Resend Eavesdropper Logic.}\label{alg:eve_intercept}
    \KwIn{List of Alice's prepared \texttt{circuits}, \texttt{interception\_rate} parameter}
    \KwOut{Modified list of circuits to be received by Bob}
    \For{each \texttt{qc} in \texttt{circuits}}{
        \If{\text{random\_value()} $< \texttt{interception\_rate}$}{
            \texttt{eve\_basis} $\leftarrow$ randomly select measurement basis ($0$ or $1$)\;
            Measure \texttt{qc} in \texttt{eve\_basis}, yielding classical bit \texttt{eve\_result}\; \Comment{Collapse the intercepted state to yield a classical bit}
            \texttt{new\_qc} $\leftarrow$ initialise empty quantum circuit\;
            Encode \texttt{eve\_result} into \texttt{new\_qc} using \texttt{eve\_basis}\; \Comment{Prepare new quantum state for resending}
            Append \texttt{id} channel marker to \texttt{new\_qc}\; \Comment{Insert identity gate to facilitate subsequent channel noise modelling}
            Replace original \texttt{qc} with \texttt{new\_qc}\;
        }
    }
    \Return{modified \texttt{circuits}}
\end{algorithm}

For computational efficiency, the implementation reflects the measurement logic of Bob, prior to constructing the replacement circuits. Critically, the replacement circuit retains the ability to model channel noise in the "resend" leg of the attack, accurately reflecting a noisy \gls{qchan} under attack.

\subsubsection*{E91 Limitations}
Whilst a noisy channel is modelled across all three protocols, the \texttt{EveInterceptor} module is constrained to the \gls{pandm} schemes. Introducing this attack model to \gls{e91} presents fundamentally different physical and simulated challenges. Because \gls{e91} relies on distributed \gls{entanglement}, an "intercept-resend" attack on one arm of the \gls{epr} pair breaks the singlet state, entirely destroying the non-classical correlation. To enact this attack upon \gls{e91}, Eve would need to execute complex \gls{entanglement}-swapping operations or deploy a sophisticated Bell-state disruption strategy \cite{E91}. 

Given the constraints of the project timeline, developing a module to perform \gls{eb} attacks was deemed out of scope. This was a deliberate decision to ensure that the implemented "intercept-resend" module reflects its theoretical basis for the \gls{pandm} protocols, without introducing inaccurate behaviours into \gls{e91}'s performance.

\section{Classical Post-Processing Pipeline}
\label{sec:postprocessing}

Following the quantum transmission phase a \gls{qkd} protocol, the system transitions into the classical domain. The post-processing pipeline evaluates the security of the distributed key material, translating quantum measurement outcomes into cryptographic metrics. Whilst physical \gls{qkd} systems would use classical algorithms for error correction (CASCADE/LDPC) and \gls{amplification}, simulating these classical protocols introduces serious computational overhead \cite{Cascade1994}. Hence, the simulator relies on information-theoretic post-processing, computing the \gls{qber}, Mutual Information bounds, and \gls{skr} under the Shannon limit \cite{Shannon1948}.

\subsection{QBER Estimation}
\label{subsec:QBER}

The \gls{qber} serves as the primary security parameter for \gls{pandm} protocols. It quantifies the discrepancy between Alice and Bob's sifted bit strings, defined as the ratio of errors ($N_{errors}$) to the total length of the sifted key ($N_{sifted}$). In a practical deployment, Alice and Bob must sacrifice a randomly selected subset of their sifted key, publicly comparing bits over the \gls{cc} to estimate the \gls{qber} of the remaining sifted key.

However, within the simulation, Alice and Bob's bit arrays are directly accessible in memory. To maximise precision, the post-processing pipeline bypasses the sacrificial sampling step and computes the \gls{qber} across the entire sifted array.

Because the \gls{sifting} logic is unique to each protocol, as described in \Cref{sec:protocolimplementation}, each scheme manages its own classical bit array extraction. Nevertheless, once the valid \texttt{sifted\_key\_alice} and \texttt{sifted\_key\_bob} arrays are isolated, the \gls{qber} calculation is executed using vectorised NumPy operations to ensure high computational throughput over large-scale samples. This logic is demonstrated in \Cref{lst:qber_calc}.

\begin{lstlisting}[language=Python, caption={Vectorised \gls{qber} Calculation Logic.}, label={lst:qber_calc}]
sifted_length = len(sifted_key_alice)

if sifted_length > 0:
    errors = np.sum(sifted_key_alice != sifted_key_bob)
    qber = errors / sifted_length
else:
    qber = 0.0
\end{lstlisting}

This \gls{qber} value is then passed to the information-theoretic analysis stage to determine whether the error rate exceeds the theoretical security thresholds, dictating whether a secure final key can be extracted.

\subsection{Mutual Information}
\label{subsec:mutualinfo}

A critical requirement of any \gls{qkd} system is determining whether Alice and Bob can actually distil a secure, shared key from their sifted data. As previously established, simulating \gls{reconciliation} and \gls{amplification} is outside the scope of this project. Instead, the simulator relies on Shannon's Information Theory to establish theoretical bounds for the key distillation process.

The fundamental metric evaluated by the post-processing pipeline is Mutual Information, which quantifies the amount of information that one random variable contains about another, as described by Johnston \cite{mutualInfo}. Because \gls{sifting} reduces the quantum states to classical bit strings, and depolarising noise applies errors symmetrically, the relationship between Alice and Bob's sifted keys can be modelled classically as a Binary Symmetric Channel (\gls{bsc}) with a crossover probability equal to the QBER. For a \gls{bsc}, the Mutual Information between Alice and Bob, $I_{AB}$, represents the capacity of the channel, accounting for the errors introduced during transmission. It is defined as a function of the \gls{qber} ($E$):

\begin{equation}
I_{AB} = 1 - H(E)
\label{eq:mutual_info_ab}
\end{equation}

where $H(x)$ is the binary Shannon entropy function, calculated as:

\begin{equation}
H(x) = -x \log_2(x) - (1-x) \log_2(1-x)
\label{eq:shannon_entropy}
\end{equation}

If $E = 0.0$ (ideal channel), $I_{AB} = 1.0$, meaning Bob possesses perfect information about Alice's bit string. Conversely, if $E = 0.5$ (depolarising noise/incorrect basis), $H(E) = 1.0$, resulting in $I_{AB} = 0.0$; Alice and Bob share no Mutual Information.

In the presence of Eve, the system must also consider Eve's Mutual Information with Alice, $I_{AE}$. Hence, to ensure unconditional security, Alice and Bob must perform \gls{amplification} to decouple Eve from the final key. Information theory dictates that a secure key can only be created if Alice and Bob share more Mutual Information than Alice shares with Eve:

\begin{equation}
I_{AB} > I_{AE}
\label{eq:secure_condition}
\end{equation}

Under the "intercept-resend" model described in \Cref{subsec:eavesdropping}, Eve's presence inherently elevates the \gls{qber} ($E$). Therefore, as attack intensity increases, $I_{AB}$ decreases whilst $I_{AE}$ increases. The intersection point where $I_{AB} = I_{AE}$ defines the theoretical security threshold of the protocol. For \gls{bb84}, this "intercept-resend" threshold theoretically sits at $E \approx 11\%$. Above this limit, \gls{amplification} cannot theoretically distil a secure key, and the transmission must be aborted \cite{ShorPreskill2000}. The simulation platform uses these equations during plotting to evaluate whether a \gls{skr} can be generated under varying noise and eavesdropping conditions.

\subsection{Security Verification: \gls{chsh} Parameter}
\label{subsec:CHSH}

Unlike \gls{pandm} schemes, \gls{e91} relies on a fundamentally different security primitive. Because \gls{e91} utilises \gls{entanglement} pairs, Alice and Bob can verify the integrity of their \gls{qchan} by testing for violations of the \gls{chsh} inequality \cite{CHSH1969}.  
 
As established in \Cref{subsec:e91}, the \gls{chsh} Bell parameter $S$ is calculated from four specific correlation coefficients $E(a_i, b_j)$, corresponding to the angle-pairs designated for \gls{chsh} estimation described in \Cref{tab:e91_sift} 

If the system exhibits perfect quantum \gls{entanglement}, $|S| = 2\sqrt{2} \approx 2.828$. However, if an eavesdropper attempts to measure the states or if the channel noise is excessively high, \gls{entanglement} is broken, reducing the system to a local hidden-variable model where $|S| \leq 2.0$.

To calculate this parameter, the pipeline logic isolates the measurement outcomes corresponding to the \gls{chsh} subset. For each of the four angle pairings, the raw binary outcomes are mapped to parity values ($+1$ or $-1$). Next, the correlation coefficient is computed as the mean of the product of Alice and Bob's parity arrays. \Cref{lst:chsh_calc} demonstrates the vectorised loop used to execute this extraction.

\begin{lstlisting}[language=Python, caption={Vectorised Extraction of \gls{chsh} Correlation Coefficients.}, label={lst:chsh_calc}]
# _CHSH_S_TERMS = [((0, 0), +1), ((0, 2), -1), ((2, 0), +1), ((2, 2), +1)]
s_value = 0.0

for (ai, bi), sign in self._CHSH_S_TERMS:
    # Isolate the subset of measurements for the angle pair
    sub_mask = (self._alice_angles == ai) & (self._bob_angles == bi)
    
    if np.sum(sub_mask) > 0:
        # Map binary (0,1) outcomes to parity (+1,-1)
        a_pm = 1 - 2 * self._alice_outcomes[sub_mask].astype(int)
        b_pm = 1 - 2 * self._bob_outcomes[sub_mask].astype(int)
        
        # Calculate expected correlation and add to total S parameter
        corr = float(np.mean(a_pm * b_pm))
        s_value += sign * corr
\end{lstlisting}

By aggregating these terms, the simulation produces a final \texttt{s\_value} for each \gls{e91} run. This value acts as the definitive security flag, where if the resultant $|S|$ fails to violate the classical bound of $2.0$ by a significant margin, the protocol aborts key distillation.


\subsection{Devetak-Winter \gls{skr} Bound}
\label{subsec:DWsec}
The final stage of the pipeline is to determine the theoretical \gls{skr}. To evaluate this uniformly across all three simulated protocols, the system employs the Devetak-Winter bound \cite{DWbound2005}.
Assuming a perfect single-photon source (as modelled by \texttt{AerSimulator}), the asymptotic \gls{skr} fraction $K$ is bounded by the difference between the Mutual Information shared by Alice and Bob ($I_{AB}$) and the maximum information an eavesdropper could theoretically extract ($I_{AE}$):

\begin{equation}
K = I_{AB} - \max I_{AE}
\label{eq:devetak_winter}
\end{equation}

As established in Section \ref{subsec:mutualinfo}, modelling the \gls{reconciliation} process is outside the scope of this simulator. Consequently, the system evaluates the theoretical maximum performance of the protocols by assuming operation at the Shannon capacity limit, meaning error correction is perfectly efficient. Under this assumption, the usable information shared between Alice and Bob is universally defined by the \gls{qber}, $E$, across all protocols: $I_{AB} = 1 - H(E)$. 

Determining Eve's penalty, $\max I_{AE}$, introduces a fundamental principle of cryptographic security proofs. In a practical \gls{qkd} deployment, Alice and Bob cannot distinguish between errors originating from environmental decoherence (noisy \gls{qchan}) and those induced by an eavesdropper. Information-theoretic security therefore requires that all channel disturbances are attributed to an eavesdropper deploying an optimal attack strategy tailored to each protocol \cite{DWbound2005}. 

Consequently, the post-processing pipeline dynamically applies a different theoretical penalty to each protocol based on its underlying physics: 

\begin{itemize}
    \item \textbf{BB84:} Eve must guess between two mutually unbiased bases. Thus, under the Shannon limit, the Devetak-Winter bound is reduced to $K = 1 - 2H(E)$, mirroring the GLLP limit \cite{GLLP2004}.
    \item \textbf{B92:} Eve must perform a USD attack. For states separated by $\theta = 45^\circ$, the penalty is: $I_{AE} = H\left(\frac{E}{1 - \cos 45^\circ}\right)$.
    \item \textbf{E91:} Eve's information is bounded by the degradation of the \gls{chsh} parameter $S$: $I_{AE} = H\left(\frac{1 + \sqrt{(S/2)^2 - 1}}{2}\right)$, adapted from Acín \textit{et al.} \cite{Acin2007}.
\end{itemize} 

This framework is implemented within the \texttt{QKDResult} base class using vectorised properties, ensuring that the final key rate is calculated consistently regardless of the protocol under test. \Cref{lst:dw_bound} details this implementation.

\begin{lstlisting}[language=Python, caption={Dynamic Application of the Devetak-Winter Bound.}, label={lst:dw_bound}]
@property
def secure_key_rate(self) -> float:
    """Calculates the Devetak-Winter \gls{skr} assuming the Shannon limit"""
    
    if self.sifted_length == 0:
        return 0.0

    # Information securely shared between Alice and Bob
    i_ab = self.mutual_information
    
    # Calculate Eve's penalty based on the specific protocol physics
    if self.protocol_name == "\gls{bb84}":
        if self.qber >= 0.110:
            return 0.0
        i_ae = self.key_rate * self._binary_entropy(self.qber)
        
    elif self.protocol_name == "\gls{b92}":
        if self.qber >= 0.065:
            return 0.0
        # Penalty assuming a USD attack
        denominator = 1.0 - (1.0 / np.sqrt(2.0))
        i_ae = self.key_rate * self._binary_entropy(self.qber / denominator)
        
    elif self.protocol_name == "\gls{e91}":
        s_val = getattr(self, 'abs_s', 0.0)
        if s_val <= 2.0:
            return 0.0
        # Penalty derived from the \gls{chsh} Bell parameter degradation
        penalty = (1.0 + np.sqrt(max(0.0, (s_val / 2.0)**2 - 1.0))) / 2.0
        i_ae = self.key_rate * self._binary_entropy(penalty)
        
    else:
        return 0.0

    return max(0.0, i_ab - i_ae)
\end{lstlisting}

If $K \leq 0$, the channel is either too noisy or compromised, and transmission is aborted. This singular metric provides the final analytical axis for evaluating and comparing protocol performance within the simulation suite.

\section{Experimental \& Statistical Methods}
\label{sec:experimental_methods}

To evaluate the protocols under varying channel conditions, the platform executes a series of benchmarking sweeps. Early in development, the system relied on single-instance protocol executions, which were highly susceptible to variance in quantum measurement. Hence, the system evolved to use a Monte Carlo approach to extract robust performance metrics.

\subsection{Central Limit Theorem and Monte Carlo Simulation}
\label{subsec:clt_monte_carlo}

As quantum measurement is inherently probabilistic, within a single protocol run, the observed \gls{qber} or \gls{chsh} parameter may significantly drift from the theoretical expectation due to random sampling noise. 

To mitigate this variance, the experiments execute $N_{trials}$ independent runs at every defined parameter point, such as a specific Noise Strength ($p$) or interception rate. By following the Central Limit Theorem \cite{montgomery2018}, the platform aggregates these independent trials to calculate the sample mean ($\mu$) and the standard error of the mean (SEM) for each metric:

\begin{equation}
\text{SEM} = \frac{\sigma}{\sqrt{N_{trials}}}
\label{eq:sem}
\end{equation}

where $\sigma$ is the sample standard deviation across the trials. This enables the results to represent statistical protocol behaviour, instead of anomalous measurements. 

\subsection{Protocol Performance Metrics}
\label{subsec:performance_metrics}

Considering the experimental variables of the simulation platform, three independent variables exist: baseline control channel, dynamic depolarising strength ($0.0 \leq p \leq 0.5$), and a dynamic interception rate ($0.0 \leq \text{Eve Rate} \leq 1.0$). 

The dependent variables, calculated in post-processing pipeline detailed in \Cref{sec:postprocessing}, include:

\begin{itemize}
    \item \textbf{QBER:} Indicates channel integrity, expressed as a percentage.
    \item \textbf{Mutual Information:} Theoretical channel capacity after \gls{sifting}.
    \item \textbf{Devetak-Winter \gls{skr} ($K$):} Cryptographic material yield.
    \item \textbf{CHSH Parameter ($|S|$):} Verifies \gls{entanglement} preservation against the classical bound ($|S| \leq 2.0$).
\end{itemize}

By correlating these metrics across identical channel noise profiles, the simulation framework provides a uniform basis for comparing the performance of the protocols.


\section{Data Visualisation}
\label{sec:dataviz}

While the simulation and statistical reduction are executed entirely within Python (as described in \Cref{subsec:clt_monte_carlo}), presenting multi-dimensional results requires dedicated rendering. Matplotlib's 3D backend is constrained by its rasterised rendering methods, preventing the precise control over surfaces required for high-quality figures. As such, the visualisation workflow is partitioned, with Python executing the simulations, then exporting the data to a structured CSV file. Finally, a dedicated MATLAB script parses this data, using the \texttt{surf()} function and \texttt{exportgraphics} to produce high-resolution 3D surface plots.

These plots evaluate the protocols against two variables: channel Noise Strength (\(p\)) and eavesdropper interception rate (\(\lambda\)). Sweeping both parameters simultaneously, reveals how these threats interact. For instance, random channel noise can hide the errors caused by Eve, or convolved interference can cause the SKR to decrease much faster than either would in isolation. Because each \((p, \lambda)\) coordinate is the mean of \(N_{\text{trials}}\) independent runs, the surfaces show reliable protocol behaviour rather than random simulation noise.

To thoroughly present the post-processing pipeline, the visualised metrics follow a logical progression from signal detection to security. The \gls{qber} is rendered first as the primary baseline, quantifying the physical degradation of the channel. To define the theoretical security boundary, the Mutual Information capacities \(I(A;B)\) and \(I(A;E)\) are plotted simultaneously on shared axes assigned distinct colours. This deliberate design choice ensures that the Devetak-Winter threshold remains distinctive.

Finally, the SKR synthesises these theoretical bounds into a single conclusive metric. For these surfaces, the colourmap is intentionally inverted to map insecure regions to the red end of the spectrum. This comprehensively presents protocol failure with a visual cliff, depicting the useful key zone without a threshold plane.

\section{Assumptions \& Limitations}
\label{sec:assumptionslimitations}
While the platform provides an environment for evaluating the cryptographic limits of \gls{bb84}, \gls{b92}, and \gls{e91}, the methodology uses several abstractions to scope the complexity of the project. To ensure the results are interpreted correctly, the following system limitations must be acknowledged:

\begin{itemize}
    \item \textbf{Shannon Capacity Limit:} As detailed in \Cref{sec:postprocessing}, modelling classical error correction (Cascade or LDPC) is outside the scope of this project. The Devetak-Winter bound assumes perfectly efficient \gls{reconciliation} at the Shannon limit. In deployment, protocol inefficiencies would yield lower key rates than the theoretical bounds produced by this simulator.
    \item \textbf{Ideal Hardware:} Qiskit's \texttt{AerSimulator} perfectly prepares single-photon states and executes unitary operations. The simulation does not model hardware imperfections, hence, the observed \gls{qber} is an isolated product of the depolarising channel and eavesdropping attacks.
    \item \textbf{E91 Eavesdropper:} As explained in \Cref{subsec:eavesdropping}, the \texttt{EveInterceptor} module is constrained to the \gls{pandm} protocols. Therefore, the performance of \gls{e91} is only evaluated against depolarising noise, while \gls{bb84} and \gls{b92} are evaluated against both noise and interception.
    \item \textbf{Perfect Classical Authenticity:} \Cref{sec:networktopology} specifies that the \gls{cc} is considered perfectly authenticated, assuming immunity to classical man-in-the-middle attacks.
\end{itemize}
